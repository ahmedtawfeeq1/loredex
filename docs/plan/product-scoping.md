# Design: Product scoping (Product → Project → Topic → Note)

**Status:** proposal — review before code. **Model chosen:** product as a grouping *attribute* on projects (one vault, no file moves). **Scope:** loredex core + desktop + Obsidian, plus a migration for the live vault.

## Problem

loredex's model is **one vault = one product**: `projects/<project>/<topic>/note.md`, and `curate --product` treats the whole vault as the product. Users with several products in one vault (e.g. GenuDo + Loredex) see projects flat, with no product layer. Desired shape is four levels:

```
Product ─▸ Project ─▸ Topic ─▸ Note
GenuDo   ─▸ genudo-ai-engine ─▸ auth ─▸ 2026-07-12-token-refresh
```

## Model: product as an attribute (not a folder)

A project belongs to at most one product. Nothing on disk moves — `projects/<project>/<topic>/` stays. The product is metadata that the **views** group by. Unassigned projects fall under an "Ungrouped" heading, so every existing vault keeps working unchanged.

### Source of truth: a committed vault manifest

Product membership must travel with the vault (teammates see the same grouping), so it lives **in the vault**, not in per-machine `~/.config/loredex/config.json`:

```
~/Loredex/_index/products.yml     # generated/managed, committed with the vault
```
```yaml
# project → product. Projects absent here are "Ungrouped".
products:
  genudo:
    - genudo-ai-engine
    - genudo-backend
    - genudo-chat-flutter
    - genudo-mcp
    - genudo-platform-front
    - genudo-website
  loredex:
    - loredex-desktop
```

This manifest is **authoritative** for the CLI Home/MOCs and the desktop tree — they read it and group, no per-note reads needed.

### Frontmatter mirror (for Obsidian only)

Obsidian's Bases plugin can only `groupBy` a frontmatter **property**, so to group the Obsidian dashboard by product, notes need `product:` in frontmatter. Plan:

- `route` stamps `product: <name>` on **new** notes (looked up from the manifest) — cheap, at write time.
- Existing notes: a one-time opt-in backfill (`loredex reindex --stamp-products`) writes the field across the vault. Not required for the desktop/CLI views (those use the manifest); only for the Obsidian "By product" view.

The manifest stays the single source of truth; frontmatter is a derived mirror.

## Changes per surface

### loredex core (npm package + CLI)

| Area | Change |
|---|---|
| `src/core/products.ts` (new) | `loadProducts(vaultPath): ProductMap`, `setProduct(vaultPath, project, product)`, `productOf(map, project)`. Read/write `_index/products.yml`. Exported from `lib.ts` for the desktop. |
| `src/commands/init.ts` | `--product <name>` writes the current project → product into the manifest. |
| `src/commands/product.ts` or new `src/commands/products.ts` | `loredex products` (list groups), `loredex products set <project> <name>`, `loredex products infer` (guess from name prefix, confirm before write). |
| `src/core/indexer.ts` | `Home.md` groups projects under `## <Product>` headings (Ungrouped last). Optional per-product MOC `_index/<product>.md`. `Dashboard.base` gains a **By product** view (`groupBy: note.product`). |
| `src/core/router.ts` (`executePlan`) | Stamp `product:` frontmatter on routed copies, from the manifest. |
| `src/commands/curate.ts` | `curate --product <name>` scopes the product dashboard/brief to that product's projects (today `--product` = whole vault; keep no-arg = all for back-compat). |
| `docs/VAULT-SPEC.md` | Document `_index/products.yml` + the `product` frontmatter field. |
| tests | products.ts round-trip; indexer grouping; route stamping; curate scoping. |

### Desktop (the view you're looking at)

| Area | Change |
|---|---|
| `src/core/tree.ts` (`walkVault`/`TreeNode`) | Insert a Product grouping level above projects: read the manifest via the new `loadProducts` lib export, wrap project nodes in product nodes. Ungrouped projects under a trailing "Ungrouped" node. |
| Reader tree UI | Renders Product ▸ Project ▸ Topic ▸ Note. Collapsible product sections (reuse the existing tree-section collapse state). |
| Settings (optional) | A "Products" pane to assign project → product from the app (writes the manifest via a new IPC → `setProduct`). Or defer to the CLI initially. |
| Home / Atlas / Search (later) | Optional product filter; not required for v1. |

Depends on loredex core shipping `loadProducts` — so the desktop dep bumps to that loredex version.

### Obsidian plugin

| Area | Change |
|---|---|
| `Dashboard.base` (generated by core) | A "By product" card view (`groupBy: note.product`) — comes for free once notes carry `product` frontmatter. |
| No plugin code change needed | grouping is data-driven by the generated base. |

## Migration (live vault — 7 projects)

1. Confirm the mapping (proposed): `genudo-*` → **genudo**, `loredex-desktop` → **loredex**.
2. Write `_index/products.yml`.
3. (Optional) `loredex reindex --stamp-products` to backfill `product:` on existing notes for the Obsidian view.
4. `rebuildIndexes` + `sync`.

Fully reversible — delete the manifest (and the frontmatter field) and it's back to flat.

## Rollout order

1. **loredex core** — manifest + `init --product` + `products` command + indexer grouping + route stamping + curate scoping → release (minor, e.g. **2.4.0**; adds `product` to the vault spec). Assign the live vault's products.
2. **desktop** — tree grouping (needs the `loadProducts` export) → new desktop release (v0.3.0).
3. **Obsidian** — the By-product base view lands with the core change; the plugin just needs a version note.

## Open questions (need your call before build)

1. **Naming** — confirm `genudo` + `loredex` as the two product names, and the 7-project mapping above.
2. **Assignment source** — explicit only (`init --product` / `products set`), or also `products infer` from name prefix with a confirm step? (Recommend: both — infer as a convenience, explicit always wins.)
3. **Backfill** — stamp `product` frontmatter on existing notes now (enables the Obsidian By-product view immediately), or only new notes going forward?
4. **Unassigned** — is an "Ungrouped" bucket fine, or should the app nudge every project to have a product?
5. **Depth in the app** — v1 = tree grouping only (your ask), or also product filters in Search/Atlas/Home?

## Effort

- Core: medium (manifest + indexer + route + curate + init + command + tests).
- Desktop: small–medium (tree grouping + optional settings pane).
- Obsidian: trivial (data-driven).
- Migration: small.

Three coordinated releases (core → desktop → obsidian), same pattern as the last cross-repo change.
